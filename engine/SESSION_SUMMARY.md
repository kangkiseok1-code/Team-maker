# 오행연 엔진 작업 요약

## 오늘(이번 대화) 한 일 한 줄 요약
엔진과 앱이 손-복사로 따로 동기화되며 자꾸 어긋나던 문제를 빌드 파이프라인으로 해결하고,
직원이 자기 성향 검사 문항 수를 고르고, 지지관계(합·충·형)와 용신 동률을 보고,
대운·세운까지 포함한 자기 만세력을 직접 볼 수 있게 만들었음.

**이전 세션과의 중요한 차이**: 직전 SESSION_SUMMARY는 "엔진 분리 직후" 상태를 적은 것이었는데,
실제로는 그 후 별도 작업(아마 다른 세션)에서 엔진 쪽에 50문항/지지관계/용신 동률 처리가
이미 구현돼 있었음. 이번 대화에서 그걸 모르고 다시 찾으려다 발견함 — **이 문서를 매번
최신 상태로 갱신해야 하는 이유가 바로 이거**.

## 1. 아키텍처: 엔진↔앱 손-복사 구조를 빌드 파이프라인으로 교체
**문제**: `ohaeng-v14.html`이 엔진 로직을 인라인으로 손-복사해서 갖고 있었음. 양쪽을 각각
고치다 보니 실제로 서로 어긋나 있었음 — 예: 앱은 용신 동률을 배열로 반환하게 고쳐졌는데
엔진(`saju.js`)은 그 전 버전(단일값)이었음.

**해결**:
- `engine/src/index.browser.js` 신규 — 앱이 실제 쓰는 모듈만 모은 브라우저용 엔트리포인트.
  (Node용 `index.js`와 분리한 이유: `daewoon.js`가 절기 데이터 278KB를 끌고 와서 무겁기 때문에,
  처음엔 빼놨다가 대운·세운 기능을 추가하면서 다시 포함시켰음 — 지금은 거의 전체와 동일)
- `engine/build.js` 신규 — esbuild로 `index.browser.js`를 `window.OhaengEngine` 전역
  하나로 번들링(`dist/ohaeng-engine.browser.js`).
- `daewoon.js`의 `fs.readFileSync`(Node 전용)를 `require()`로 바꿔서 브라우저 번들링 가능하게 수정
  (동작은 동일, 데이터를 디스크에서 읽던 것 → 번들에 내장).
- `ohaeng-v14.html`의 첫 `<script>` 안 내용을 이 번들로 통째로 교체. 앱의 두 번째 `<script>`는
  `const { computeSaju, ... } = OhaengEngine.saju;` 식으로 엔진에서 가져다 쓰고,
  로직을 손으로 다시 안 베낌.
  - `pairSynergy`만 예외: 엔진 시그니처가 `pairSynergy(memberA, memberB)`로 바뀌어 있어서,
    기존 호출부(인덱스 `i,j`로 부름)를 안 건드리려고 얇은 래퍼 함수 하나만 앱에 남겨둠.
- 검증: `node --test`(엔진 69개), `node --check`(앱 문법), **jsdom으로 실제 페이지를 띄워서
  로드 에러·핵심 플로우(사주 계산·궁합·지지관계·관리자 상세 카드 렌더)까지 확인**.

**남은 수동 단계**: `node build.js` → 번들 파일을 `ohaeng-v14.html`에 붙여넣는 건 아직 스크립트
하나로 안 합쳐져 있었음 → **이번에 `engine/update-app.js`로 합침** (아래 6번 참고).

## 2. 발견: 엔진엔 이미 있었지만 SESSION_SUMMARY엔 안 적혀있던 것들
- `personality.js`: `getPersonalityQuestions(20|50)` — 이미 구현돼 있었음.
- `synergy.js`: `pairBranchRelations(A,B)` — 합·충·형만(파·해는 유파차로 제외) 점수에 안 섞고
  설명만 반환하는 함수, 이미 구현돼 있었음.
- 둘 다 **엔진엔 있었는데 앱(`ohaeng-v14.html`)에는 아직 안 옮겨져 있었음** → 이번에 앱에 포팅.

## 3. 이번에 새로 고친 엔진 버그
- `saju.js`의 `deepAnalyze`: 용신이 동률일 때 단일값만 반환하던 걸 **배열로 반환**하게 수정
  (앱에는 이미 이 수정이 들어가 있었는데 엔진엔 없어서 둘이 어긋나 있었음 — 엔진을 앱 기준으로 맞춤).
  - 1986-09-17 08:30 검증 케이스에서 목·수 동률 확인(`["목","수"]`).
- `daewoon.js`: `fs`/`path` 의존 제거(위 1번 참고).

## 4. 앱에 새로 포팅/구현한 기능
- **성향 20/50문항 선택**: emp-input에 토글 추가(기본 20문항). `buildSurveySequence`의 보조문항
  끼워넣는 간격을 문항 수에 비례하게 일반화(`count/10`개마다 하나) — 20문항일 땐 기존과 100% 동일,
  50문항이어도 끝까지 고르게 섞임.
- **시너지에 지지관계 표시**: 궁합 페어 카드에 합·충·형 설명을 점수 없이 추가. 파·해는 제외.
- **관리자 상세 카드 버그 수정**: 용신이 배열이 됐는데 관리자 화면(`memberDetailHtml`)만 단일값
  가정이라 깨져있던 걸 수정 + 성별 표시 추가.
- **emp-result(최초 결과화면)**: 내정보 탭과 내용이 사실 이미 같았음(공유 렌더 함수). 다만
  뒤로 가서 생년월일·성별을 고칠 방법이 없어서 "← 정보 다시 입력" 링크만 추가.
- **내 만세력(대운·세운) — 신규**: emp-result·내정보 양쪽에 "내 만세력" 섹션 추가.
  - 대운 8구간 표(나이대, 간지, 그 대운만으로 원국과 합·충·형이 생기는지 태그 표시)
  - 연도 입력 → 그해 세운 + 해당 대운까지 합친 관계(합·충·형, 점수 없이 설명만)
  - 시각·성별이 없으면 계산 대신 안내 문구만 표시(대운 계산에 둘 다 필요)
  - **관리자/팀보드(`memberDetailHtml`)에는 의도적으로 노출 안 함** — "직원이 데이터 소유" 원칙대로
    팀보드는 기존 정보 수준만 유지(기석님 명시적 요청).

## 5. 오픈소스 비교 후 발견한 갭 해소 (야자시, 월운·일운)
다른 오픈소스 명리 도구(rath/orrery, sajupy 등)와 기능을 비교해보다가 두 가지 빈 곳을 발견해서 채움:

- **야자시(夜子時)**: 기존엔 23시생이어도 그날 일주를 그대로 썼음(조자시 방식만 가능했음).
  이제 `computeSaju(...,{yajasi:true})` 옵션으로 23시생은 다음날 일주로 계산할 수 있게 함
  (시주 천간도 새 일간 기준으로 같이 재계산됨 — 만세력 라이브러리의 `getGapja`에 다음날 날짜만
  넘기면 나머지는 라이브러리가 일관되게 계산해줌). **유파차 영역이라 기본값은 끔(조자시,
  기존 동작 보존)** — 앱에 토글로 노출, 켜고 끄면 결과가 실제로 바뀌는 것까지 확인.
- **월운·일운**: 세운(연)·대운(10년)보다 짧은 주기가 없었음. `transit.js` 신규 모듈로 "그 날짜
  자체의 일주·월주"를 뽑아서(생년월일과 무관, `getGapja(연,월,일)` 그대로 사용) `timeline.js`의
  `combinedTimeline()`에 같이 합쳐볼 수 있게 함. 그 김에 `combinedTimeline`이 대운·세운 외에도
  임의의 추가 운(`extraPillars` 배열)을 받을 수 있게 일반화함(기존 3개 인자 호출부는 그대로 호환).
- 앱의 "내 만세력" 연도 입력을 연/월/일 입력으로 확장해서 세운+월운+일운+대운을 한 화면에서
  같이 보여줌.
- 엔진 테스트 75개로 증가(전부 통과), jsdom으로 야자시 on/off 결과가 실제로 달라지는 것·관리자
  상세 카드엔 여전히 대운/세운/월운/일운/야자시 단어가 전혀 없는 것까지 화면 단위로 재확인.

## 6. 빌드 자동화
`engine/build.js`(번들링)와 "그 번들을 ohaeng-v14.html에 끼워넣기"가 따로따로였던 걸
`engine/update-app.js` 하나로 합침. `node update-app.js [html경로]` 한 줄로 번들링→교체→문법검증
까지 끝남. 손으로 만든 결과와 byte 단위로 동일한 것, 두 번 연속 실행해도 안전한 것(idempotent)
확인함.


- jsdom으로 실제 `ohaeng-v14.html`을 `runScripts:'dangerously'`로 띄워서, 폼에 값 채우고
  버튼 클릭까지 시뮬레이션해 화면 단위로 회귀 확인(엔진 단위 테스트로는 못 잡는, "앱에 제대로
  연결됐는지"를 확인하는 용도).

## 7. UI 재설계 — 레이더 통합, 간지 카드(사진 참고), 형살(刑) 강조
기석님이 실제 쓰는 다른 만세력 앱 캡처를 참고해서 두 화면을 다시 디자인함.

- **12운성 포팅**: 엔진엔 있었지만 브라우저 엔트리에 없던 `twelve-stages.js`를 `index.browser.js`에
  추가. 참고 캡처(1986-09-17 08:30 차트)로 사주 네 기둥 12운성(연주=건록·월주=태·일주=목욕·
  시주=쇠)과 십신(식신·상관·본원·편재)이 전부 정확히 일치하는 것까지 확인.
- **내정보(mypage) 그래프 통합**: 오행 분포 바·빅파이브 바·십신 바·가치관 바, 이렇게 4군데
  흩어져 있던 막대그래프를 팀보드에 있던 오각형 레이더 차트 하나로 합침. "오행·십신·성향·가치관"
  4개 버튼으로 같은 차트를 전환(전부 0~100% 척도로 맞춰서 탭 전환 시 안 튐). emp-result("두 개의
  나" 비교 화면)는 좌우 비교가 의도된 레이아웃이라 그대로 둠 — 내정보만 바꿈.
- **간지 카드로 네 기둥 재설계**: 참고 캡처처럼 천간·지지를 오행 색상 박스(한자+한글)로, 위에
  십신, 아래에 지장간(본기만 밑줄 표시 — "지금 어느 지장간이 활성인지"는 절기 진행도 계산이
  필요해서 아직 안 함, 추측 표시 안 함)과 그 지장간의 십신, 12운성까지 한 카드에 표시
  (`gzCardHtml()`). 원국·대운·세운 전부 같은 카드 함수를 재사용.
- **대운·세운을 가로 스크롤 카드 띠로**: 기존 세로 목록을 참고 캡처처럼 가로로 스크롤되는 카드
  띠로 바꾸고, 지금 나이대/올해는 회색 배경(`now` 클래스)으로 표시.
- **형(刑) 강조**: 대운·세운 카드가 원국과 합쳐졌을 때 삼형·형(부분)·자묘형·자형이 생기면 그
  카드에 빨간 테두리 + "⚠ 형 종류" 배지를 띄움(`hyeongTypesFor()`). 원국 자체에 형이 있는 경우도
  네 기둥 카드에 똑같이 표시됨.
- jsdom으로 사진 속 실제 수치 일치, 인사신·축술미 삼형 인위 케이스로 배지 표시, 탭 전환 시
  레이더 데이터가 실제로 바뀌는 것까지 화면 단위로 검증.

## 8. 데모 버전 워크플로 확립
기석님 요청으로 "내정보·팀보드에 항상 사주 10명이 채워진 데모"를 매 업데이트마다 같이 만들기로
함(`memory_user_edits`에 등록해둠 — 다음 세션에서도 유지됨). 가상 인물 10명을 시드 고정 난수로
생성하는 `gen-demo-members.js`(엔진으로 실제 계산, 손으로 채운 가짜 숫자 아님)로 `ohaeng-v14.html`의
`state` 초기값에 주입하고 `myIdx=0`(첫 번째 인물)을 기본 지정해서, 내정보·팀보드 둘 다 바로 보이게
함. 10명 중 한 명(최유나)이 마침 원국 자체에 삼형이 걸려 있어서 형살 강조 기능의 좋은 예시로 쓰임.


```
engine/
  src/
    index.js              Node용 전체 엔트리(모든 모듈)
    index.browser.js      브라우저용 엔트리 — daewoon·sewoon·timeline·transit·twelveStages 포함
    saju/daewoon.js        fs/path 의존 제거(브라우저 호환)
    saju/saju.js           deepAnalyze 용신 동률 배열 처리 + computeSaju에 yajasi 옵션 추가
    saju/transit.js        일운·월운(특정 날짜의 일주·월주)
    saju/timeline.js        combinedTimeline에 extraPillars(월운·일운 등) 인자 추가
    saju/twelve-stages.js   (기존 모듈, 이번에 브라우저 엔트리에 추가만 함)
  build.js                 esbuild로 index.browser.js를 OhaengEngine 전역으로 번들링
  update-app.js            build.js + ohaeng-v14.html에 번들 끼워넣기를 한 번에
  dist/ohaeng-engine.browser.js  빌드 결과물(git에 안 넣어도 됨, build.js로 재생성 가능)
```

별도(엔진 zip 바깥, 대화에서 따로 전달): `gen-demo-members.js` — 데모용 가상 인물 10명 생성 스크립트.
